# Mobile Invoice Form

## Purpose

Tracks the Expo mobile invoice/quote form under `apps/expo-app/src/features/sales/invoice-form`.

## Current Architecture

- The form uses shared `@gnd/sales/sales-form-core` helpers for workflow line pricing, grouped row patches, HPT door summaries, shelf row patches, record hydration, and save payload generation.
- `@gnd/sales/sales-form-core` is treated as the native-safe sales-form boundary for Expo; it must not export from web UI TSX files or any module importing `@gnd/ui`, and this boundary is covered by `packages/sales/src/sales-form-core.native-safety.test.ts`.
- Expo app source must not import the web UI package directly; `apps/expo-app/src/features/sales/invoice-form/native-ui-boundary.test.ts` scans mobile source for `@gnd/ui` and direct `packages/ui` references so mobile remains native-owned while sharing only pure sales-domain helpers.
- Shared sales-form workflow helpers use one domain metadata parser for object-or-JSON metadata shapes, keeping mobile reopen behavior consistent without duplicating parser code across domain and workflow modules.
- Mobile save/edit/update flows remain routed through the invoice form store:
  - `actions.patchLineItem(...)`
  - `actions.buildSavePayload(...)`
  - `toSalesFormSaveDraftPayload(...)`
  - `newSalesForm.saveDraft` / `newSalesForm.saveFinal`
- Mobile save/finalize requests treat the persisted invoice/quote record as the user-facing completion boundary. Server-side document snapshot expiration, inventory sync queueing, and document warmup queueing run as bounded best-effort post-save tasks so slow document/cache infrastructure or running mobile without the jobs worker cannot leave the mobile form stuck on `Saving invoice...`; the mobile client also sends mutations over an unbatched tRPC link, bounds the save await path, and drives the overlay from store `saveStatus`, so hung requests surface a retryable footer error instead of permanently blocking the form.
- Dev-mode invoice save debugging now captures parsed `newSalesForm.saveDraft` / `saveFinal` payloads on the API server before the core save transaction. Captures are written under `debug/new-sales-form-save-payloads/YYYY-MM-DD/*.json` at the project root with summary metadata plus the full payload, are ignored by Git, run only when `NODE_ENV === "development"`, and never fail the save if the file write fails.
- The mobile invoice form store parses line metadata through the shared sales-form parser before matching existing source lines or propagating taxable state into grouped service rows, so JSON-string metadata does not break native line merging or grouped row updates after reopen.
- The shared sales form schemas preserve rich grouped payloads with pass-through fields for:
  - `formSteps`
  - `shelfItems`
  - `housePackageTool.doors`
  - grouped row metadata such as `meta.serviceRows` and `meta.mouldingRows`

## Mobile Step Editors

Specialized invoice line editors now live under dedicated step-family folders:

```text
apps/expo-app/src/features/sales/invoice-form/steps/
  house-package-tool/
  moulding/
  service/
  shelf-items/
  shared/
```

`line-item-card.tsx` composes these editors instead of owning every grouped row UI inline.

## Behavior Notes

- The mobile sales dashboard `New Invoice` action uses the reusable standalone `FloatingFooterActionChooser` pattern to show a native floating footer Sales/Quote chooser without `@gorhom/bottom-sheet` or the shared modal wrapper, includes a list-row Cancel action, resets the in-memory create form, applies the selected type, and then routes to the typed customer-selector route before entering the invoice form; Sales maps to `order` mode and Quote maps to `quote` mode through a focused helper with coverage, while the separate dispatch shortcut remains admin-only. New, edit, and customer-selector invoice routes share one native route-param normalizer for the sales form type.
- Initial mobile invoice/quote creation no longer mounts the customer selector as a separate native stack route before replacing it with the invoice route. The invoice form renders `CustomerStep` inline until a customer is selected, then reveals the item workflow in the same screen tree, avoiding Android edge-to-edge native view removal during customer selection.
- The invoice item swipe/workflow wrapper uses stable React Native `style` for its fixed minimum height instead of the arbitrary NativeWind class `min-h-[640px]`, avoiding `react-native-css` variable-set remount warnings during the customer-to-items transition.
- Android sales-form helper routes for customer selection, sales details, and door-size entry use ordinary card presentation while keeping the headerless full-screen visual shape; iOS keeps native full-screen modal presentation. This avoids Android edge-to-edge `react-native-screens` modal traversal crashes when customer selection immediately replaces the selector route with the invoice route.
- Android edge-to-edge is disabled in `apps/expo-app/app.config.ts` after the crash continued inside `EdgeToEdgeReactViewGroup`; testing this mitigation requires installing a fresh Android native build, not only reloading Metro or applying an OTA update.
- The mobile sales dashboard uses a structured loading skeleton instead of a centered spinner, preserving the page's stat-card, action-row, and recent-sales-card rhythm while the overview query is pending.
- The mobile sales dashboard now exposes a live `Quotes` entry beside `Orders`. The quotes route reuses the mobile orders list architecture for back navigation, search, filters, pull-to-refresh, empty state, and document cards while querying `sales.quotes` and `filters.salesQuotes`; quote cards show quote-specific status (`Paid`, `Open`, `Part paid`) from invoice pending/total amounts and open the existing quote-aware invoice edit form when the quote has a saved slug.
- The mobile sales dashboard Recent Sales section renders the shared `SalesDocumentCard` order card used by the Orders list instead of a bespoke compact row, mapping the dashboard recent-sales payload into the card's invoice total/paid/due, customer, phone, date, and delivery-option fields.
- New invoice route params, form initialization, save labels, and the customer selector preserve the selected sales form type through the shared route-param helper.
- The dedicated mobile customer-selector route keeps customer results as a virtualized list while moving search into a fixed bottom input powered by `react-native-keyboard-controller` `KeyboardStickyView`, so the search control stays reachable and rises above the keyboard. The inline initial customer step inside the invoice form keeps its header search layout.
- Details edit/read-only, workflow notice, local-recovery, item-selector, and item-sheet surfaces use quote-aware labels for the screen title, details section heading, document reference number, line-item helper copy, sparse item fallback titles, recovery banner, quote date, good-until date, save/create actions, and notes/memo copy instead of leaking invoice-only wording.
- Customer/item selector helper text, item list actions, item sheet labels, and sparse item fallbacks use quote-aware wording when the selected sales form type is `quote`; the customer selector and customer list receive the route type directly so first render does not flash invoice-only copy or query the wrong recent-customer type. Customer selector, review, and details surfaces share sparse-customer display fallbacks so recent/search rows do not render dangling contact separators or blank billing copy when phone or billing address data is missing.
- Create/reset state starts with no mock/default selected customer; customer state is set only by explicit customer selection, selected-customer bootstrap hydration, recovered local payloads, or an existing saved record.
- Fresh create/reset/bootstrap state starts with no mock/default line item before customer selection, but explicit customer selection seeds the first blank workflow line when the form has no items so the original inline `Item Type` step pills and component grid appear immediately after the customer handoff. Empty create-mode bootstrap hydration preserves that selected customer's seeded workflow line, or reseeds it when the bootstrap already carries a selected customer, so the post-customer route handoff cannot clear the first item. The items screen groups the seeded first line as a workflow section keyed by line uid, keeping it active for the inline workflow selector and its step pills. The empty-state Add item action remains available for itemless states, while shared save validation rejects saving without a line item.
- Create-mode form bootstrap receives the selected customer id from the customer selector handoff, so server bootstrap data cannot overwrite the chosen customer with an unscoped default.
- The customer step defaults to 10 unique recent customers for the selected type, grouped by customer and ordered by each customer's latest order/quote timestamp; the mobile request shape is centralized in a focused helper so blank order/quote customer screens always request `recent: true`, the selected type, and `limit: 10`, while typed search text switches the list to explicit search results. The API overfetches recent customer groups before hydration so missing stale customer references do not under-fill the default list.
- Workflow component cards, workflow-root item selector rows, created/hydrated workflow line-card titles/subtitles, inline item description inputs, review item titles, workflow step selection labels/pills, Door Size picker headers, HPT door/group labels, HPT add/swap door option chips, custom component option labels, workflow selectable search rows, and newly added workflow-line descriptions no longer show UID text or UID-like workflow slugs as visible copy or title/SKU/value/category fallbacks; UID remains available for identity, keys, routing, search matching, and state. ## Validation Guidance

- Manual mobile QA should cover create and edit saves for mixed invoice lines containing HPT doors, service rows, moulding rows, and shelf items.
- Lightweight static validation is preferred by default in this Bun monorepo; broad Expo typecheck/build/browser validation should be run only when explicitly requested.
