# Dyke + Inventory Integration Status And Roadmap

## Status
Analysis with implementation slices applied

## Created Date
2026-06-16

## Scope
Review whether the current GND Dyke + inventory integration supports:
- invoice create/update syncing inventory sales lines
- produceable/non-produceable Dyke behavior in inventory demand
- production assignment/completion syncing inventory projections
- save-time inbound status updates feeding the correct inbound dashboards
- production/fulfillment completion updating inventory sales state
- inventory sales modeling for useful analytics

## Verdict
The inventory-backed sales model is substantially implemented, but it is not fully cut over or fully proven. Invoice save paths now queue inventory sales line sync, and the inventory model has useful sales-line, component, allocation, inbound, and analytics structures. Produceable/non-produceable semantics are partially applied: inventory sync records production eligibility, and production plan/readiness ignores non-produceable inventory-backed lines while leaving fulfillment data intact. Legacy production mutation bypasses are also largely collapsed by calling the inventory production lifecycle bridge from direct app-level assignment/submission/delete/mark-complete paths. The inbound prompt now updates existing open line-level demand statuses for `ORDERED` / `PENDING ORDER`, later sales-inventory sync projects the saved order status onto newly created demand, and the inbound dashboard exposes prompt-vs-demand reconciliation. The API/package boundary and modal surface now support selected `InboundDemand` ids for line-level prompt refinement, allowing selected pending/ordered demand to be cancelled when those specific lines are confirmed `AVAILABLE` without making the order-wide `AVAILABLE` prompt destructive. ADR-009 documents inventory as the owner of inbound demand status semantics, with the pure status and selected-demand mutability policy exported through `@gnd/inventory/inbound-policy` for app/UI consumers and `@gnd/inventory/inbound` for server/package consumers. The remaining main gaps are fixture-backed mutating browser validation, save-time prompt proof from invoice save through `/inventory/inbounds`, and final reconciliation/backfill proof.

## Roadmap

### Phase 1: Seal Produceable Semantics
- Completed first slice: `sync-sales-inventory-line-items` derives and stores production eligibility, and `buildSalesProductionPlan` skips non-produceable lines.
- Completed first slice: non-produceable but stockable lines can still keep inventory component data for fulfillment/backorder analytics because the filter lives in the production projection, not the component sync.
- Completed proof slice: explicit non-produceable metadata wins over legacy Dyke production flags when deriving inventory production eligibility.
- Completed proof slice: `buildSalesBackorderQueue` keeps non-produceable stock demand visible in fulfillment queues while production planning ignores the same lines.
- Completed proof slice: persisted mixed grouped metadata rows keep HPT and service rows produceable while keeping moulding rows non-produceable, and HPT metadata still extracts HPT/door inventory candidates.
- Remaining: browser/runtime proof from real saved mixed grouped orders and review of any route-config production flags not represented in sales item metadata.
- Remaining: confirm whether inventory dashboards should expose a "non-produceable inventory demand" filter for stock-only lines.

### Phase 2: Collapse Production Bypasses
- Completed first slice: `batch-assign-production-orders`, legacy assignment/submission/delete actions, old mark-complete actions, sales-progress fallback, clean-code production data-access helpers, mirrored `app-deps` helpers, and older production item actions now call `syncInventoryProductionLifecycleForSale` after direct mutations.
- Completed first slice: `apps/www/src/actions/production-control-reset.test.ts` guards direct production mutation files for the bridge call.
- Completed proof slice: the canonical `update-sales-control` production lifecycle sync decision now lives in `@gnd/sales` control application code and is regression-covered for all production mutations, while dispatch-only mutations remain excluded.
- Remaining: fixture/browser proof that each user-facing production action refreshes inventory metadata in real workflows.

### Phase 3: Unify Inbound Status With Inventory Demand
- Completed first slice: keep `SalesOrders.inventoryStatus` as the order-level human prompt while updating existing open `InboundDemand` rows to `pending` or `ordered` when the prompt is `PENDING ORDER` or `ORDERED`; `PENDING ORDER` does not downgrade partially received or shipment-linked demand.
- Completed first slice: sales inventory line sync now uses the same inventory-owned demand status resolver when it creates demand rows after the order prompt has already been saved.
- Completed first slice: `/inventory/inbounds` now shows a reconciliation panel for orders with prompt/demand mismatches, extracted as a dedicated widget to keep the workspace page more Midday-style and composition-oriented.
- Completed first slice: `notes.saveInboundNote` / `applyOrderInboundStatusToInventoryDemand` accept selected `demandIds` for line-scoped mutation; selected `AVAILABLE` cancels only selected mutable demand, while broad `AVAILABLE` remains non-destructive.
- Completed first slice: the inbound status modal fetches active mapped demand for the open sale, uses the inventory-owned `canOrderInboundPromptMutateDemand` helper for selected-row enablement, and can submit selected `demandIds` for line-scoped prompt updates.
- Completed proof slice: invalid non-empty selected-demand id lists are skipped at the inventory boundary instead of falling back to order-wide demand mutation.
- Remaining: browser-validate the selected-demand modal flow and save-time invoice prompt path with mapped demand fixtures.
- Remaining: browser-validate real save-time prompt flows and ensure mapped demand appears correctly in the inbound dashboard.

### Phase 5: Analytics And Reconciliation Gate
- Backfill inventory-backed `LineItem` rows for real historical sales.
- Run the sales inventory sync monitor and reconciliation report.
- Add dashboards for unmapped legacy sales, skipped inventory mappings, stale inventory sale lines, and stale allocations/demands.
- Completed first slice: the inventory import control center now displays review-risk counts plus bounded componentless sale and stale inventory sale-line samples from the sales inventory sync monitor.
- Completed first slice: stale inventory sale-line cleanup is available through a dry-run-default inventory tRPC mutation, and the import control center can clean the currently visible stale samples after confirmation.
- Completed first slice: `inventories.salesInventorySyncMonitor` can opt into a bounded dry-run reconciliation summary, keeping the default query light while letting cutover callers fold inventory reconciliation drift into the package-owned review-risk gate.
- Completed monitor gate slice: a clean-but-partial bounded reconciliation run now keeps the sales inventory sync monitor in `needs_review` until the cursor is exhausted, so partial proof cannot be read as cutover-ready sync.
- Completed monitor gate slice: skipped reconciliation comparisons also keep the reconciliation summary in `needs_review` and contribute to review-risk count through `skippedComparisonCount`, so unverified shipment/allocation or component-fulfillment rows cannot pass as clean proof when drift is zero.
- Completed report evidence slice: raw inventory reconciliation reports and queued Trigger run output now expose shared status plus total skipped-comparison count, and the import control center completion summary names the status and skipped comparison rows when a queued reconciliation finishes.
- Completed UI slice: the inventory import control center now requests the bounded reconciliation summary and surfaces it in the existing sales sync monitor card plus the system checks list, preserving the Midday-style dashboard composition while making Phase 5 drift visible to operators.
- Completed UI signal slice: the Review Risk stat subtitle now names componentless sales, stale inventory lines, reconciliation drift, and skipped reconciliation comparisons so the headline risk count explains its package-owned inputs.
- Completed UI action slice: the inventory import control center now queues the existing `run-inventory-reconciliation-report` Trigger task from the sales sync monitor section, reusing run-status polling and starting from the current reconciliation cursor when the bounded summary reports `hasMore`.
- Completed UI detail slice: when reconciliation is requested, the import control center now renders compact domain coverage cards with checked, drift, skipped, severity, skipped-reason, and sample counts so operators can tell whether the risk is sales sync, shipment/allocation, component fulfillment, or incomplete/skipped comparison coverage.
- Completed monitor slice: the sales inventory sync monitor now reports stale stock-allocation and inbound-demand residue attached to stale inventory sale lines, and the import control center shows those counts in the Review Risk stat.
- Completed UI check slice: stale stock-allocation and inbound-demand residue now has its own system-check row, making stale fulfillment residue a visible cleanup gate before analytics cutover.
- Promote inventory-backed analytics only when coverage and caveats are visible to operators.
