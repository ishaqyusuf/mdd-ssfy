## Highest-Risk Findings

- P1 - Contractor job and payroll reads are still public at the tRPC boundary.
  Evidence: `apps/api/src/trpc/routers/jobs.route.ts:509` through `535` expose `getJobForm`, `getJobs`, `paymentDashboard`, `paymentPortal`, `contractorPayouts`, `contractorPayoutOverview`, and `getContractorPayoutPrintData` as `publicProcedure`. The backing queries select sensitive operational and payroll data: `apps/api/src/db/queries/jobs.ts:179` through `229` selects job notes, amounts, status, contractor/user names, admin notes, payment ids, project and unit context; `apps/api/src/db/queries/jobs.ts:611` through `642` reads payable contractors; `apps/api/src/db/queries/jobs.ts:648` through `741` returns payment counts, amounts, recent payments, contractor names, and payer names; `apps/api/src/db/queries/jobs.ts:1430` through `1500` can build all-unpaid payroll print data with contractor emails and job amounts. Inference: even if the app UI is permission-gated, unauthenticated API reads can leak contractor payroll, job assignments, and builder/project context.

- P1 - HRM directory and permission reads remain public after quick-login hardening.
  Evidence: `apps/api/src/trpc/routers/hrm.route.ts:24` safely returns an empty quick-login list, but `getEmployees`, `getDrivers`, `getRoles`, `getProfiles`, and `getPermissions` remain `publicProcedure` at lines `25` through `64`. Brain permissions now state mobile quick login should use the empty dedicated route, but these broader reads still expose employee and permission catalog data. Inference: this weakens operational privacy and makes the disabled quick-login list less meaningful because adjacent employee-list routes are still callable without a session.

- P2 - Active Sessions gives a false sense of account control.
  Evidence: `apps/www/src/components/user-logged-in-devices.tsx:9` through `23` renders hard-coded mock devices, and `logOutDevice` at lines `35` through `39` waits 500ms before clearing state with `TODO: Implement actual logout logic.` The Brain feature file `.brain/features/settings-profile-table.md` confirms this placeholder behavior is known. For office/admin users, this is a trust and security-accountability gap: they see device rows and a Log out action, but no real session is inspected or revoked.

- P2 - Storefront catalog projections still risk showing internal cost or wrong variant pricing.
  Evidence: `packages/sales/src/storefront.ts:205` says cost prices still need transformation by default customer profile, and `apps/api/src/trpc/routers/storefront.route.ts:498` selects the first variant when multiple variants are available. The Storefront Public Launch Readiness plan already requires production-safe pricing/parity signoff before launch. Inference: customer-facing price filters/cards can be misleading for configurable doors, mouldings, and shelf items if the projection uses raw cost or the wrong variant.

- P2 - Inventory correctness remains not release-clean.
  Evidence: `.brain/tasks/in-progress.md` records the July 23 read-only evidence refresh as `needs_backfill` with `21745` missing sales, `50` stale lines, `5` componentless lines, `281` reconciliation drifts, `14` skipped comparisons, `hasMore=true`, and next cursor `200`. Repairs are stopped by user request. This remains a production readiness risk for inventory allocation, production planning, and fulfillment confidence.
