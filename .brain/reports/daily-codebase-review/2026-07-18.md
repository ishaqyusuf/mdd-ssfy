## Highest-Risk Findings

### P1 - Public high-impact operational mutations remain visible across API routers

Evidence:

- `apps/api/src/trpc/routers/dispatch.route.ts` imports `publicProcedure` and exposes operational mutations including `deletePackingItem`, `cancelDispatch`, `startDispatch`, `submitDispatch`, `updateSalesDeliveryOption`, `updateDispatchDriver`, `updateDispatchDueDate`, `updateDispatchStatus`, and `uploadDispatchDocument` through public procedures.
- `apps/api/src/trpc/routers/hrm.route.ts` exposes employee mutations such as `resetEmployeePassword`, `deleteEmployee`, `revokeEmployee`, `restoreEmployeeAccess`, `setEmployeeBugReportingAccess`, and `saveEmployee` as public procedures.
- `apps/api/src/trpc/routers/inventories.route.ts` exposes high-impact inventory operations including `resetInventorySystem`, category save, inventory form, and inventory save actions through public procedures.

Impact: dispatch status, proof documents, employee access, and inventory state need clear authenticated role boundaries for accountability. Even if some downstream checks exist, the route contract itself is inconsistent and difficult to reason about, train, or audit.

Inference: this is not a claim that every route is externally exploitable in production; it is a route-boundary risk because the reviewed router code does not make the intended permission contract explicit.

### P1 - Sales document Share can send tokenized customer PDFs to a hard-coded phone number

Evidence:

- `apps/www/src/components/sales-menu.tsx` generates a tokenized sales PDF share URL with a 7-day expiry, then calls `share({ url, msg, recipient: "+234 8186877306" })`.

Impact: a sales rep using the live Share action can send a customer/order document link to a fixed phone number rather than the actual customer, dealer, or selected contact. This is a direct customer-trust and privacy risk for quotes, invoices, receipts, and production/order documents.

Brain coverage: Plan exists in `.brain/plans/2026-07-02-feature-sales-document-whatsapp-sms-delivery.md`, but the current hard-coded recipient should be treated as a bug-level blocker inside that plan.

### P1 - Dealer quote approval is still bypassable or ambiguous through old conversion/edit surfaces

### P2 - Mobile dispatch proof upload and completion are client-orchestrated, not atomic

Evidence:

- `apps/expo-app/src/features/dispatch/components/dispatch-complete-form.tsx` collects signature and optional photo attachments.
- `apps/expo-app/src/features/dispatch/api/use-dispatch-documents.ts` uploads proof documents one-by-one through `dispatch.uploadDispatchDocument`.
- `apps/expo-app/src/features/dispatch/components/dispatch-detail-screen/index.tsx` uploads attachments, uploads signature, then separately calls `actions.onSubmitDispatch`.

Impact: if a field worker loses network during completion, uploaded proof and dispatch completion can diverge. Workers with varying technical comfort need a single clear completion state, not a sequence that can partially succeed without a durable recovery path.

Brain coverage: adjacent mobile reliability plans exist, but this exact dispatch proof atomicity slice appears to be a new candidate.

## Worker Usability Review

Sales reps still face too many document delivery states that are easy to misuse. A Share action that silently uses a fixed recipient is not trainable; the worker needs explicit recipient confirmation, channel choice, send status, and an audit trail.

Production and inventory staff need inventory readiness phrased as operational proof, not an engineering caveat. Brain correctly keeps repair work paused, but the UI and release process should keep the cutover state visible until reconciliation is clean.

Warehouse, driver, and installer workflows need recoverable mobile actions. Dispatch completion should show whether proof upload, status completion, and customer/dealer notification are each complete, pending, or failed.

Managers need route-level accountability for actions that change employee access, inventory, dispatch, and customer-facing documents. Public route declarations make the intended ownership model hard to inspect.

## Customer And Dealer Experience Review

The dealership quote-to-order flow is conceptually right: dealers request conversion and internal staff approve. The remaining bypass/edit paths create a mismatch between what dealers are taught and what the software still allows.

Customer document trust is fragile. Quotes, receipts, invoices, dispatch proof, and production order documents should always be delivered to the customer/dealer contact on the record or to a deliberately selected recipient.

Dealer dashboard clarity should show request state, internal owner, last revision, and whether a quote is locked, needs changes, approved, rejected, or converted. This reduces calls and prevents dealers from editing values that internal staff already approved.

## Door-Manufacturing Workflow Gaps

Quote readiness should expose door-specific checks before approval or production handoff: dimensions, hinge side, swing, profile, species/material, glass/lite requirements, jamb/casing assumptions, finish, hardware preparation, and delivery/pickup method.

Production readiness should link quote/order documents, inventory allocation, payment state, and dispatch promise into one checklist that sales and production can both read.

Inventory allocation visibility should distinguish quoted demand, approved order demand, production-reserved materials, picked stock, packed stock, and dispatched stock. Current Brain evidence shows the correctness foundation is still in progress.

Mobile proof should support photo proof, signature proof, barcode/QR checks where practical, and a clear retry state for poor warehouse or field connectivity.
