## Highest-Risk Findings

- P1 - Internal operational routes still expose too much through `publicProcedure`. Evidence: `apps/api/src/trpc/routers/dispatch.route.ts:75` defines the dispatch router with public list, mutation, upload, bulk, delete, and restore paths; `apps/api/src/trpc/routers/inventories.route.ts:629` onward exposes many inventory admin/import/mutation paths publicly; `apps/api/src/trpc/routers/hrm.route.ts:23` onward exposes employee/admin mutations publicly. Some individual procedures check `ctx.userId`, but the route boundary is inconsistent for workflows that affect delivery status, stock, employee access, and customer trust.
- P1 - Dealer direct quote conversion remains live beside the approval workflow. Evidence: `apps/api/src/trpc/routers/dealer-portal.route.ts:398` exposes `convertQuoteToOrder`, while `requestQuoteOrder` starts at line 403 and `.brain/features/dealership-quote-to-order-approval.md` says direct conversion should be replaced by sales-rep approval. This can confuse dealer training and risks bypassing the reviewed delivery-cost/approver ownership path if any UI or client still calls the old mutation.
- P1 - Mobile dispatch proof completion is not atomic or clearly resumable. Evidence: `apps/expo-app/src/features/dispatch/components/dispatch-detail-screen/index.tsx:1193` uploads photo attachments, line 1211 uploads signature SVG, and line 1221 then submits dispatch completion. If upload succeeds but completion fails, the worker receives only "Unable to complete dispatch"; if completion succeeds after a retry, duplicate proof/attachment handling is unclear. This is high impact for installers, drivers, warehouse staff, and customer delivery disputes.
- P1 - Full repo validation remains blocked. Evidence: `packages/documents/src/index.ts:1` uses extensionless NodeNext exports, and `packages/documents/src/types.ts:9` references `Buffer`; `bun run typecheck` stops at `@gnd/documents`. Document infrastructure is shared by dispatch proof, bug reports, sales PDFs, and future document migration, so this blocks confidence across customer-facing proof and print flows.
- P2 - Sales document Share still sends tokenized PDFs to a fixed phone number. Evidence: `apps/www/src/components/sales-menu.tsx:571` generates a tokenized share URL and line 585 passes `recipient: "+234 8186877306"`. This is now covered by the WhatsApp/SMS plan, but remains a customer privacy and sales-rep trust issue while live.
- P2 - Dealer approval delivery-cost capture still uses `window.prompt`. Evidence: `apps/www/src/components/dealer-request-review-banner.tsx:20` and `apps/www/src/components/sales-rep-dealer-requests.tsx:60`. The backend protected approval path is good, but prompt-based cost entry is easy to mistype, hard to explain to office staff, and weaker than a structured modal with delivery type, destination, ZIP, and approval note context.

## Worker Usability Review

Mixed-skill users need fewer implicit states and fewer one-off controls. The active tables-2 migration improves scannability for office, dispatch, inventory, HRM, and community tables, and the Sales Overview Inventory tab has more explicit locked states. The remaining rough edges are approval prompts, generic mobile timeout/error copy, and route-level public/protected inconsistency that makes it hard to reason about who is accountable for a write.

Mobile field/warehouse flows are better than earlier runs because dispatch completion now requires a signature and supports photos, and invoice recovery exists. The weak point is transaction shape: workers should not need to understand whether an upload, packing action, signature conversion, notification, or completion mutation failed separately.

## Customer And Dealer Experience Review

Dealership has strong recent coverage: dealer customers default private, direct-ship snapshots are immutable, approval/payment/customer/internal PDFs were browser-proven, and dealer dashboard metrics are real. The remaining customer/dealer trust gaps are old direct conversion still existing, prompt-based delivery-cost entry, and source document sharing still using a hard-coded recipient in the main web sales menu.

Customer-facing print/PDF architecture is in better shape through Sales PDF V2, but document package typecheck and shared document migration remain blockers for confidence in uploads, snapshots, and future proof records.
