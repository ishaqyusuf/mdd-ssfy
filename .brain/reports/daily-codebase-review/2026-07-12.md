## Executive Summary

Today's review found the same operational release risks as the recent daily runs, with stronger evidence around dealership and dispatch proof flows.

The top risk remains high-value operational API boundaries. Several mutation groups that affect dispatch, HRM, settings, notes/documents, and sales operations still use `publicProcedure`; some handlers do local `ctx.userId` checks, but many do not have an adjacent permission gate. That is hard to train, hard to audit, and risky for office/admin teams because actions like dispatch cancellation, driver assignment, employee role/profile changes, setting updates, document uploads, and delete/restore flows should have explicit accountability.

Dealer quote request state is still not a hard edit boundary. The dealer quote table disables repeat "Request order" actions after a request is pending/approved/rejected, but the same row still exposes `Edit`, and the save path updates the dealer-owned quote by id/type without checking request status. A dealer could change dimensions, profiles, or pricing after requesting approval, which undermines sales-rep review and customer trust.

## Highest-Risk Findings

- P1 - Operational mutations still have weak or inconsistent route boundaries. Evidence: `apps/api/src/trpc/routers/dispatch.route.ts:75` defines the dispatch router with `publicProcedure` for start/cancel/submit/status updates, packing slip signing, dispatch creation/delete, document upload, bulk assignment/cancel, export, restore, and duplicate resolution. `apps/api/src/trpc/routers/hrm.route.ts:22` exposes employee password reset, delete/revoke/restore, bug-reporting access, save, profile update, and role update through `publicProcedure`. `apps/api/src/trpc/routers/settings.route.ts:71` exposes general `updateSetting` publicly while app-download settings use a protected Super Admin gate. `apps/api/src/trpc/routers/user.route.ts:60` exposes employee document save/upload/review/delete and notification preference updates publicly. Inference: downstream helpers may reject some unauthenticated calls, but the route surface itself does not make permission intent reviewable.

- P1 - Dealer requested quotes remain mutable after request/approval state. Evidence: `apps/dealership/src/components/tables/quotes/columns.tsx:102` computes pending/closed request state and disables only `Request order`, while `apps/dealership/src/components/tables/quotes/columns.tsx:115` always exposes the `Edit` link. `apps/dealership/src/components/dealer-sales-form/dealer-quote-composer.tsx:867` wires save actions for edited quotes. `packages/db/src/queries/dealers.ts:2903` loads an existing dealer quote only by id/dealer/type/deleted state, then `packages/db/src/queries/dealers.ts:2962` updates it without checking `DealerSalesRequest` status. The request path in `packages/db/src/queries/dealers.ts:3132` only prevents duplicate pending requests.

- P1 - Inventory-backed fulfillment is still not safe to claim release-ready. Evidence: Brain tracks the live cutover as not clean in `brain/tasks/in-progress.md`, `brain/tasks/roadmap.md`, and `brain/plans/2026-07-01-inventory-system-correctness-cutover-plan.md`: `20449` missing sales, `9` shipment/allocation drift, `1` skipped comparison, `hasMore=true`, and repairs stopped by user request. ## Worker Usability Review

- Sales reps need confidence that approved/requested dealer quotes cannot silently change after review. Today the UI state says "Requested" but still leaves edit affordances available.
- Production and warehouse workers need inventory readiness to be visibly authoritative. Current Brain state says inventory is still not clean, so any "ready" messaging should remain qualified until the cutover reaches clean reconciliation and browser/operator proof.
- Drivers and installers get retry UI on dispatch load failure, which is good. The completion path still assumes online upload success before final submit and does not appear to offer a durable pending-proof queue.
- Admin staff need a simple permission story. Public route names mixed with inline auth checks make it difficult to teach which actions are safe and who is accountable.

## Customer And Dealer Experience Review

- Dealer quote-to-order is close to the right model, but the edit-after-request gap can create a customer-facing mismatch between what the dealer requested, what the sales rep approved, what the customer sees on PDF/payment, and what production receives.
- Sales document WhatsApp/SMS delivery is already planned and remains a practical customer-experience improvement, especially for quote acceptance and payment links.
- Dealer payment-link creation checks due amount and wallet identity, which is good, but the broader approval/payment handoff should stay tied to immutable approved quote revisions.

## Door-Manufacturing Workflow Gaps

- Quote approval should lock the manufacturing-relevant quote revision: dimensions, hinge/swing, profile, supplier, finish, tax/profile pricing, delivery mode, and PO/payment terms.
- Production readiness should be based on clean inventory reconciliation plus explicit exceptions for non-stock/untracked components.
- Dispatch completion should store proof as one server-owned transaction: packed state, signature, photos, receiver, note, location/time when available, and status transition.
- Inventory allocation should keep manager-visible drift exceptions separate from normal worker queues so staff do not learn to ignore "ready" labels.
