## Highest-Risk Findings

### P1 - Inventory fulfillment remains not release-clean, with repairs explicitly stopped

Evidence: `brain/tasks/in-progress.md` and `brain/plans/2026-07-01-inventory-system-correctness-cutover-plan.md` both state the latest Phase 8 evidence is still `needs_backfill` with `20449` missing sales, `9` shipment/allocation drift rows, `1` skipped comparison, `hasMore=true`, and next cursor `208`. They also state no further repair dry-runs or applies should run unless Ishaq explicitly resumes repairs.

Operational risk: sales, warehouse, purchasing, production, and dispatch can see partially proven inventory truth. The current state is correctly controlled by Brain, but it is not ready for a broad cutover claim.

### P1 - Shared document package blocks full typecheck

Evidence: `bun run typecheck` fails in `@gnd/documents`; `packages/documents/src/index.ts` exports extensionless NodeNext imports, and `packages/documents/src/types.ts` references `Buffer` without available Node types.

Operational risk: documents are central to quotes, invoices, dealer/customer PDFs, dispatch signatures, attachments, and proof workflows. A broken shared document package narrows confidence in any change touching customer trust or paper/PDF handoff.

### P1 - Dealer-submitted quotes appear mutable after request/approval state

Evidence: the dealer quotes table disables only `Request order` for `pending`, `approved`, and `rejected` requests, but leaves `Edit` available in `apps/dealership/src/components/dealer-portal/dealer-sales-documents.tsx`. The server save path in `apps/api/src/db/queries/dealer-portal-sales-form.ts` updates any dealer-owned `type: "quote"` by id, resets `status` to `Draft`, deletes existing line items, recreates them, and updates dealer totals without checking the latest `DealerSalesRequest` status.

Operational risk: a dealer could change line items, pricing, tax, delivery option, or totals after asking GND to approve the quote, or after a sales rep has approved/rejected it. That creates disagreement between the sales-rep decision, customer-facing document, payment amount, and production handoff.

### P2 - Dealer payment link path can dead-end on missing wallet

Evidence: `dealerPortal.createPaymentLink` only creates a checkout link after finding an existing customer wallet; otherwise it throws "This customer does not have a payment wallet yet." The dealership feature goal says approved dealers should receive and open payment links.

Operational risk: approved dealer orders can still require office intervention if a customer identity exists but wallet creation has not happened. For mixed-skill office teams, that is a confusing handoff because the visible CTA says payment is available.

### P2 - Mobile dispatch proof is present, but not warehouse-scan oriented

Evidence: mobile dispatch completion supports signature and attachment upload via `apps/expo-app/src/features/dispatch/components/dispatch-complete-form.tsx`, and the dispatch detail screen uploads those files before `submitDispatch`. The packing and dispatch flows do not show barcode/QR scan confirmation for item/order verification in the reviewed paths.

Operational risk: photo/signature proof helps after delivery, but warehouse workers still rely on visual selection for packing correctness. Door manufacturing would benefit from scan-confirmed pick/pack/dispatch steps for large, similar-looking door and trim items.

## Worker Usability Review

Sales reps have a strong Brain-backed path toward inventory-aware sales overview, inbounds, and document delivery. The risk is that several controls are still in "planned polish" rather than reliable daily workflow: disabled Create inbound / Mark available behavior, inbound counts that depend on tab activation, and inventory proof gates that remain open.

Dealership users get a simple quotes/orders flow, which is good for adoption, but the submitted-quote lifecycle needs a clearer mental model: draft quotes should be editable; requested/approved quotes should either lock or create an explicit revision that tells GND what changed.

Warehouse and drivers have mobile packing, signature, and attachment capabilities. The next usability leap should be fewer ambiguous taps: scan order/line/item, show exactly what is expected, and require explicit completion proof only where it matters.

Office/admin staff still need typecheck and document reliability stabilized before document-platform migration or multi-channel send work expands.

## Customer And Dealer Experience Review

Dealer/customer trust depends on quote totals, PDFs, approval status, and payment links matching. The current dealer quote edit/save path can break that trust after a quote is requested. Payment links can also fail if no wallet exists, even when the dealer order overview shows a Pay balance CTA.

Customer communication is well-covered by the proposed WhatsApp/SMS plan. It should stay high priority because door customers and builders often operate around phone-first updates, not email-only document delivery.

## Door-Manufacturing Workflow Gaps

- Inventory correctness still has open reconciliation and operator proof gates before it can be relied on as production/purchasing truth.
- Dealer quote approval needs immutable submitted snapshots or an explicit revision flow before sales reps review delivery cost and approve.
- Mobile warehouse packing would benefit from barcode/QR confirmation for order, line, and item/variant identity.
- Dimension, swing, hinge, profile, and supplier-specific door pricing checks are tracked across new sales form parity and inventory supplier pricing work, but remain spread across multiple workstreams.
- Sales document delivery beyond email is planned and should be tied to short links and audit/activity records.
