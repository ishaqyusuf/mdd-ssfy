# Daily GND Codebase Review - 2026-07-15

## Executive Summary

Today's review found one meaningful improvement and the same highest-risk release blockers. The active dealership work narrows the dealership app's local tRPC route to `dealershipAppRouter` with only `dealerPortal` and `google`, which reduces the earlier risk of the dealer deployment tracing the full internal API router. That improvement does not close the broader authorization concern because high-value operational routes in the main API still use `publicProcedure` for dispatch, HRM, settings, notes, sales, inventory, jobs, community, and customer mutations.

The riskiest operational gap remains dealer quote mutability after a quote-to-order request starts. Dealer UI disables duplicate `Request order` actions for pending/approved/rejected requests, but still renders `Edit`, and `saveDealerPortalQuote` updates any dealer-owned quote by id without checking the current request state. For door manufacturing, this can change dimensions, line items, pricing, delivery terms, or customer profile data after the office has started approval or payment handoff.

Inventory correctness is still not release-clean by Brain evidence. The active cutover remains blocked by `needs_backfill`, `20449` missing sales, `9` shipment/allocation drift rows, `1` skipped comparison, `hasMore=true`, and next cursor `208`. Repairs remain stopped by user request, so this automation did not run repair dry-runs, applies, migrations, data syncs, or evidence commands.

Broad typecheck failed today in `@gnd/documents` with NodeNext extensionless-import errors, missing `Buffer` type availability, and implicit provider callback parameter types. That keeps shared document confidence low for quotes, invoices, dealer documents, dispatch proof, employee documents, and future WhatsApp/SMS document delivery.

## Highest-Risk Findings

### P1 - Operational API route boundaries are still inconsistent

Evidence: `apps/api/src/trpc/routers/dispatch.route.ts` still exposes mutating dispatch routes as `publicProcedure`, including `submitDispatch` at line 181, `updateDispatchStatus` at line 201, `deleteDispatch` at line 369, `uploadDispatchDocument` at line 424, `bulkAssignDriver` at line 457, and `bulkCancel` at line 462. Broader static search shows many public procedures across HRM, notes/notification channels, settings, sales, inventory, jobs, community, and customer routes.

Positive change: `apps/api/src/trpc/routers/dealership-app.ts` now scopes dealership to `dealerPortal` and `google`, and `apps/dealership/src/app/api/trpc/[...trpc]/route.ts` uses that router instead of the full internal `appRouter`.

Operational risk: mixed-skill office, production, warehouse, and driver users need clear accountability. Public route boundaries make it harder to prove who can mutate delivery, inventory, employee, settings, or customer data.

### P1 - Dealer requested quotes can still be edited after request state changes

Evidence: `apps/dealership/src/components/tables/quotes/columns.tsx:116` and `apps/dealership/src/components/dealer-portal/dealer-sales-documents.tsx:122` still render `Edit`; request buttons are disabled for pending/approved/rejected statuses at lines 150 and 127-129 respectively. Server-side, `apps/api/src/db/queries/dealer-portal-sales-form.ts:355` finds any dealer-owned quote by id, then `salesOrders.update` at line 418, deletes line items at line 439, and upserts dealer totals at line 467 without a visible request-state guard.

Operational risk: a dealer can revise a quote after sales approval work begins, weakening delivery-cost review, payment-link trust, customer invoice consistency, and production-readiness review.

### P1 - Inventory release readiness remains blocked

Evidence: `.brain/features/inventory-backed-sales-fulfillment.md` and `.brain/tasks/in-progress.md` state the current reconciliation is not clean: `needs_backfill`, `20449` missing sales, `9` shipment/allocation drift, `1` skipped comparison, `hasMore=true`, next cursor `208`. Repairs are stopped by user request.

Operational risk: sales, production, purchasing, warehouse, and dispatch should not treat inventory-backed fulfillment as release-clean until missing-sales scope and shipment/allocation drift are resolved or explicitly product-scoped.

### P2 - Dispatch proof upload is not atomic with completion

Evidence: mobile dispatch completion uploads attachment blobs at `apps/expo-app/src/features/dispatch/components/dispatch-detail-screen/index.tsx:1196`, uploads signature SVG at line 1211, then calls `actions.onSubmitDispatch` at line 1221. The upload route is `publicProcedure` at `apps/api/src/trpc/routers/dispatch.route.ts:424`.

Operational risk: a driver on weak network can upload some proof, fail final completion, and leave proof separated from the completed dispatch action. That is a customer trust and accountability risk for delivery/pickup workflows.

### P2 - Shared documents still block broad validation

Evidence: today's `bun run typecheck` failed in `@gnd/documents` before full workspace validation completed.

Operational risk: documents sit under quotes, invoices, PDFs, dealer/customer documents, dispatch proof, employee documents, bug recordings, and planned WhatsApp/SMS delivery.

### P2 - Dealership deployment is unblocked by skipping build-time type validation

Evidence: `apps/dealership/next.config.mjs:4` sets `typescript.ignoreBuildErrors: true`; `.brain/progress.md` records that `bun --filter @gnd/dealership build` passed while skipping type validation and that dealership typecheck still reports baseline errors.

Operational risk: this is a pragmatic deployment unblock, but not a release-confidence substitute for the dealer portal. Quote/order/payment surfaces still need focused type and browser proof.

## Worker Usability Review

- Sales reps need a hard visual and server boundary between editable dealer drafts and office-reviewed dealer requests. The current UI teaches "Requested" for conversion, but still leaves "Edit" available.
- Production and warehouse teams benefit from the inventory workbench direction, but Brain still says proof gates are open for operator/browser evidence. Training should avoid promising inventory truth until the cutover is clean.
- Drivers/installers need dispatch completion to feel like one action: capture proof, submit, and get a durable retry if the network drops. The current client-orchestrated upload sequence is understandable, but fragile.
- Office/admin staff benefit from simplified task monitor work; the new diagnostics ledger is already implemented in Brain, but deployment confidence depends on the database and typecheck baselines staying visible.

## Customer And Dealer Experience Review

- Dealer-facing router scoping is a good trust improvement because dealership deployments no longer need the full internal API router surface.
- Dealer quote request status is visible, but edit availability after request state makes the customer/dealer experience ambiguous. A dealer may think a revised quote is the reviewed version even if office approval was based on older terms.
- Payment-link and print flows are directionally strong, but document typecheck failure keeps PDF/document surfaces a release-readiness concern.
- Customer communication improvements are tracked through sales email ledger and WhatsApp/SMS plans; do not expand channels until document and route boundaries are cleaner.

## Door-Manufacturing Workflow Gaps

- Requested dealer quotes need a revision/lock model: changed dimensions, swing, hinge, profile, moulding, HPT, shelf, delivery option, or price after request should either be blocked or become a new revision needing office review.
- Inventory-backed production readiness still needs clean reconciliation and operator proof before managers can rely on it for purchasing, backorder, production, and dispatch decisions.
- Dispatch proof should bind photos/signature/received-by metadata to the final completion transaction or a durable completion draft/retry bundle.
- A manager-facing readiness view should combine dealer pending requests, inventory reconciliation status, document/typecheck status, failed document sends, and dispatch proof failures so risks are visible before customers feel them.
